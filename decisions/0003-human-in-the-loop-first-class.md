# 3. Human-in-the-loop как штатный режим, не fallback

**Status:** Accepted
**Date:** 2026-05-09
**Authors:** Кирилл Копылов

## Context

KZ marketplaces (OLX, Kaspi Объявления) активно защищены анти-бот системами (Cloudflare, JS challenges, фингерпринты). Простой HTTP-запрос возвращает страницу-челлендж, не листинги. Playwright + stealth-плагины помогают, но **не панацея** — часть запусков всё равно блокируется. Похожая ситуация на части других источников.

При этом KZ marketplaces — наш самый ценный источник для редких выгодных сделок (P2P listings). Игнорировать их или принимать частые BLOCKED-ответы — терять смысл всего продукта.

В классической архитектуре scraper'а блокировка → лог `BLOCKED` → продолжаем дальше. Это «потеря данных» как штатное поведение.

## Decision

**Human-in-the-loop встроен как первоклассный режим работы, не как fallback.**

Когда автоматика не справляется:
1. Скрипт `fetch_kz.py` **не пишет** `BLOCKED` и не идёт дальше
2. Возвращает `{"status":"needs_human", "reason":"cf_challenge", "url":"..."}`
3. Агент добавляет задачу в Sheet `Pending_Help_Queue` со status=`WAITING_FOR_HUMAN`
4. Шлёт Telegram-уведомление категории 🆘 HELP_NEEDED с конкретной инструкцией
5. Запуск завершается с exit code 0 (не ошибка — корректное поведение)

Когда пользователь готов помочь:
1. Запускает `bash help-deals.sh`
2. Открывается interactive Claude Code сессия
3. Агент через `claude-in-chrome` MCP открывает реальный Chrome
4. На каждом барьере (капча/cf-challenge/login) **просит пользователя пройти руками** и ждёт подтверждения
5. После прохождения парсит страницу, извлекает листинги, пишет в Sheet
6. Обновляет задачу на `DONE` и присылает summary в Telegram

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| Просто логировать `BLOCKED` и пропускать | Теряем самые ценные данные. Пользователь не узнаёт что упустил выгодную сделку. |
| Captcha-solving сервис (2captcha, Anti-Captcha) | Дорого ($1-3 за капчу × 5+ блокировок в день = $5-15/день = $150-450/мес). Этический серая зона — некоторые ToS запрещают. |
| Residential proxy без HITL | $50-100/мес, не решает капчи на 100%, добавляет complexity. Может комплементировать HITL но не заменяет. |
| Полная автоматизация через mouse/keyboard эмуляцию (computer-use MCP) | Хрупко, банят agents, эскалирует противостояние с площадками. |
| Отказаться от scraping, переходить на B2B data partnerships | Невозможно на pre-seed/seed стадии — нет рычага для контрактов с marketplaces. |

## Consequences

### Positive
- **Ноль потерянных запусков** — каждое намерение проверить источник либо доходит до конца, либо ждёт человека
- Нравственно чисто — мы не обходим anti-bot, а просим живого человека (которому площадка явно разрешает доступ)
- IPO-friendly — отсутствие captcha-bypass / aggressive anti-bot circumvention в коде
- Pending_Help_Queue даёт прозрачную метрику «насколько автоматика работает» (% задач которые ушли на ручное закрытие)
- Архитектура естественно деградирует и восстанавливается — если Mac выключен и не пришёл human, задачи копятся; когда Mac включён, человек закрывает их пакетно

### Negative
- **Зависимость от пользователя.** Если человек уехал в отпуск без internet — задачи копятся, потом много обработки.
- Cognitive load на пользователя — постоянно нужно «открыть Claude Code и закрыть очередь»
- Telegram-уведомления могут быть triggering / annoying если приходят часто. Mitigation: monitor «частоту HELP_NEEDED», если выше N в день — считать риском (R-08 в risk register)
- Сложнее объяснять инвесторам — не обычная архитектура. Может выглядеть «не масштабируемым».

### Neutral
- Параметр для оценки PMF: насколько пользователю не лень закрывать очередь. Если копится — это сигнал что value не достаточен.
- Потенциальная feature: «Ask community to help» — другие пользователи могут закрыть твою очередь за reward. Пост-Series A.

## Revisit triggers

- **Pending_Help_Queue >50% от total runs стабильно** — автоматика не справляется, нужны другие меры (proxy, partnership, отказ от sources)
- **Founder выгорание от уведомлений** — если фаундер сам игнорит alerts → user тоже будет
- **Series B-C scaling:** к этой стадии нужно > 90% автоматизации, HITL только для edge cases
- **Captcha технология эволюционирует** (e.g., браузеры начнут блокировать sites которые её используют) — может быть unblock'ом
- **Партнёрские data feeds** становятся доступны — заменяем scraping на API
